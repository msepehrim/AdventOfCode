Pythonic Refactor

list_all = []
list_operator = []

with open("day06_input.txt") as infile:
    for line in infile:
        parts = line.split()
        # Numeric line
        if parts and parts[0].isdigit():
            nums = [int(p) for p in parts if p.isdigit()]
            list_all.append(nums)
        # Operator line
        else:
            ops = [p for p in parts if p in {"+", "*"}]
            list_operator.extend(ops)

col_count = len(list_all[0])

grand_total = 0

for col_idx, op in enumerate(list_operator[:col_count]):
    column_values = [row[col_idx] for row in list_all]

    if op == "+":
        sub_total = sum(column_values)
    else:  # op == "*"
        sub_total = 1
        for v in column_values:
            sub_total *= v

    grand_total += sub_total

print(grand_total)

ChatGPT said:

Great question and the answer is important for understanding list structure in Python.

Why extend() instead of append()?

Because ops is a list of operators, not a single operator.

Example:

python
ops = ["+", "*"]

If you use:

python
list_operator.extend(ops)

You get a flat list:

python
["+", "*"]

But if you use:

python
list_operator.append(ops)

You get a nested list:

python
[["+", "*"]]

Your later code expects the operator list to look like:

css
["+", "*", "+", ...]

not:

css
[["+", "*"], ["+", "+", "*"], ...]

More Concretely

Your computation loop does:

python
for col_idx, op in enumerate(list_operator[:col_count]):

This expects each element of list_operator to be a single operator, e.g.:

arduino
"+" or "*"

But if you used append(), each element would be a list:

css
["+","*"]

so op would be a list, and the code would break.

Summary

GoalCorrect method
Add each string individuallyextend()
Add the list as one itemappend()

Since we want a flat list of operators, extend() is correct.
